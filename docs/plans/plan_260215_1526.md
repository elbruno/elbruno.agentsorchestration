# Implementation Plan: Interactive Multi-Turn Console Experience

**Date:** 2026-02-15 15:26  
**Status:** ✅ COMPLETED

## Overview

Implemented a **multi-turn, interactive console experience** that allows users to:

1. Generate code via orchestration
2. View generated files
3. Run the generated application
4. Discuss further iterations and apply changes incrementally
5. Maintain full conversation context across turns

## What Was Implemented

### 1. Core Models: Conversation Session Records

**File:** `src/ElBruno.AgentsOrchestration.Orchestration/Orchestration/Models.cs`

Added two immutable records to the Orchestration project:

- **`ConversationTurn`** — Captures a single turn: user prompt, orchestration result, timestamp
- **`ConversationSession`** — Tracks:
  - `SessionId` — unique identifier
  - `WorkspacePath` — associated workspace
  - `History` — immutable list of all turns
  - `CreatedAt` — session timestamp
  - `BuildContextPrompt()` — method to build multi-turn context for agents
  - `AddTurn()` — immutable update method using `with` expressions

**Design:** Follows the library's pattern of sealed immutable records. No persistence layer (that's left to integrations).

### 2. Conversation Manager Service

**File:** `src/ElBruno.AgentsOrchestration.Core/Orchestration/ConversationManager.cs`

Implements:

- **Session lifecycle** — Create, retrieve, record turns, clear sessions
- **Thread-safe dictionary** — Tracks active sessions via lock
- **Context building** — Sessions can build full conversation context for multi-turn prompts

**Use case:** Integrations (console, Aspire, etc.) use this to maintain conversation state without reimplementing the logic.

### 3. Enhanced Interactive Console Sample

**File:** `samples/ConsoleCompleteChat/Program.cs`

Complete redesign from single-run to interactive loop:

**Main Menu** (when no session active):

```
[1] Start new orchestration
[2] List active sessions
[3] Exit
```

**Conversation Menu** (when session active):

```
[1] Add feature / make changes          → Multi-turn orchestration with context
[2] Run the app                         → LaunchAsync via AppRunner
[3] View generated files                → Browse workspace
[4] Start new orchestration             → Switch to new session
[5] Exit
```

**Multi-Turn Flow:**

1. User enters initial prompt → orchestration runs → workspace created → session initialized
2. Session recorded in `ConversationManager`
3. User selects [1] to add features → prompted for next change
4. Session rebuilds context: `"Conversation so far:\n[User]: ...\n[Assistant]: ...\n\n[User]: {newChange}"`
5. New orchestration with full context → agents make incremental changes
6. Turn recorded in session history

**Features:**

- ✅ Graceful Console.Clear() handling for piped input
- ✅ Real-time event logging with emoji indicators
- ✅ AppRunner integration for launching generated apps
- ✅ File browser for inspecting workspace
- ✅ Session list display
- ✅ Conversation turn tracking

## How It Works: The Flow

### Initial Run

```
User: "Create a .NET console app that prints Hello World"
     ↓
OrchestrationService.RunAsync()
     ↓
WorkspaceManager creates timestamped directory
     ↓
ConversationSession created → AddTurn(prompt, result)
     ↓
Display menu with options
```

### Multi-Turn (Add Feature)

```
User: "Add a feature that displays the current date"
     ↓
Session.BuildContextPrompt(userMessage)
     ↓
"Conversation so far:
[User]: Create a .NET console app...
[Assistant]: Orchestration verified...

[User]: Add a feature that displays the current date"
     ↓
OrchestrationService.RunAsync(contextPrompt)
     ↓
Agents see full history → make incremental changes
     ↓
session.AddTurn(userMessage, result) → history updated
     ↓
Display menu again
```

### App Execution

```
User: "Run the app"
     ↓
AppRunner.LaunchAsync(workspacePath)
     ↓
dotnet run --no-build in workspace directory
     ↓
Stream stdout/stderr to console in real-time
     ↓
User presses Enter → Stop the app
```

## Design Decisions

### ✅ Conversation State in Core, Not Orchestration

- `ConversationSession` lives in **Orchestration** (immutable models)
- `ConversationManager` lives in **Core** (service layer)
- This keeps Core focused on orchestration while providing conversation utilities
- Aspire dashboard can optionally use these same utilities if desired

### ✅ No Persistence by Default

- Sessions are in-memory only (stored in `ConversationManager` dictionary)
- No file I/O, database, or JSON serialization in Core
- Integrations can layer persistence on top if needed
- Simplifies testing and avoids complexity

### ✅ Context Building in ConversationSession

- Each session knows how to build its own context prompt
- Format: `"Conversation so far:\n[User]: ...\n[Assistant]: ...\n\n[User]: {latest}"`
- Matches Aspire's current behavior
- Agents receive full history and can make intelligent incremental changes

### ✅ No Changes to OrchestrationService

- Service is unchanged — just passes the built context prompt
- Agents don't need to know about conversation tracks; they just see a prompt
- Clean separation of concerns
