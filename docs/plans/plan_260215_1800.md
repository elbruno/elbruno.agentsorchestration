# Plan: Researcher Agent with Inter-Agent Communication and Flow Tracing

**Date:** 2026-02-15 18:00  
**Status:** Not Started  
**Complexity:** High  
**Estimated Effort:** 3-4 days

## Overview

Implement a new **Researcher** agent that acts as a specialized research assistant for other agents in the orchestration pipeline. The Researcher will leverage specific Copilot tools (web search, Microsoft Learn MCP, Context7 MCP) to gather information when other agents need external knowledge. The Orchestrator will be enhanced to detect research needs and coordinate inter-agent communication, with full tracing and visualization of agent call flows.

**Architectural Foundation**: All agents in the system (Orchestrator, Planner, Coder, Designer, Fixer, BuildReviewer, and Researcher) use **GitHub Copilot Agent** powered by **Microsoft Agent Framework** as their core intelligence. This is the out-of-the-box behavior, not a future enhancement. The `TemplateAgentClient` exists solely for deterministic testing in CI/CD pipelines.

## Goals

1. **New Researcher Agent Role**: Add a new agent that specializes in information gathering using web search and documentation sources
2. **Inter-Agent Communication**: Enable agents to request research from the Researcher and receive enriched context
3. **Orchestrator Enhancement**: Update Orchestrator to route research requests intelligently and support non-sequential flows (loops, retries, dynamic delegation)
4. **Flow Tracing**: Implement comprehensive tracing of inter-agent calls as a core library feature (enabled by default)
5. **Flow Visualization Data**: Generate visualization data (Mermaid diagrams, JSON flow graphs) that consuming applications (Console, Web, API) can render

  - Example markers: `[RESEARCH_NEEDED: query]` or `[REQUEST_RESEARCH]` tags
  - Heuristic detection: questions about unknown libraries, recent changes, best practices
  
- **Retry and Loop Support**:
  - Track failure counts per agent/step
  - Implement configurable retry policies (max attempts, backoff)
  - After N failures, trigger alternative paths (e.g., request research)
  
- **Dynamic Delegation Flow Example**:
  1. Builder fails (attempt 1)
  2. Orchestrator invokes Fixer
  3. Builder fails again (attempt 2)
  4. Orchestrator invokes Fixer with more context
  5. Builder fails third time (attempt 3)
  6. Orchestrator delegates to Researcher: "Find solutions for this build error"
  7. Researcher returns research response
  8. Orchestrator invokes Fixer with research-enriched context
  9. Builder succeeds (attempt 4)
  
- **State Management for Loops**:

  ```csharp
      public int MaxRetries { get; init; } = 3;
      public int ResearchThreshold { get; init; } = 3; // Trigger research after N failures
      public bool ResearchUsed { get; set; }
  }
  ```

### 4. Flow Tracing and Telemetry (Core Library Feature)

**Core Principle**: Flow tracing is a built-in feature of the orchestration libraries, not specific to any consuming application. The tracing system produces structured data that can be consumed by Console apps, Web dashboards, APIs, or any other client.

**Default Behavior**: Tracing is **enabled by default**. Applications can opt-out when creating an orchestration or during execution via configuration.

**Non-Sequential Flow Support**: The tracing system must handle complex flows including:

- **Loops**: Build failures triggering multiple Fixer attempts
- **Conditional Research**: After N failed attempts, Orchestrator requests Researcher to find solutions online
- **Dynamic Routing**: Orchestrator decides next agent based on previous results
- **Parallel Execution**: Multiple agents working simultaneously (future enhancement)

Example complex flow:

```
Orchestrator → Planner → Coder → Builder → [FAIL]
                                   Builder → [FAIL] 
                                   Builder → [FAIL]
                                   Orchestrator → Researcher (search for build error solution)
                                   Researcher → Orchestrator
                                   Orchestrator → Fixer (with research context)
                                   Fixer → Builder → [SUCCESS]
```

## Risks and Mitigations

**Risk**: Microsoft Agent Framework API may differ from assumed design  
**Mitigation**: Abstract tool configuration behind `IAgentClient` interface; research Agent Framework documentation early in Phase 1; adjust implementation as needed

**Risk**: MCP server integration may require additional setup/authentication  
**Mitigation**: Review MCP server documentation; implement authentication layer if needed; provide clear setup instructions

**Risk**: Inter-agent communication adds complexity to orchestration pipeline  
**Mitigation**: Keep communication protocol simple (request/response); add comprehensive logging and event streaming

**Risk**: Flow visualization performance issues with many agent calls  
**Mitigation**: Limit visualization to last N calls; use virtualization in UI; optimize Mermaid generation

**Risk**: Detection of research needs may have false positives/negatives  
**Mitigation**: Use explicit markers initially; refine heuristics based on usage; allow manual override
