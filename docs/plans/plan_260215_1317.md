## Goal

Restructure `AgentsOrchestration.Core` into focused, independently publishable NuGet packages that the .NET community can reuse to build multi-agent orchestration systems with the Microsoft Agent Framework.

## Analysis

The original `AgentsOrchestration.Core` project mixed three distinct concerns:

1. **Agent abstractions** вЂ” foundational types (roles, configuration, interface, factory) needed by anyone building any agent system
2. **Orchestration engine** вЂ” the "Ultralight Orchestration" pipeline pattern (plan в†’ parse в†’ execute в†’ verify в†’ report)
3. **Infrastructure** вЂ” workspace file management, process launching, template implementations

These concerns have different audiences and reuse patterns:

| Concern | Who uses it | How often it changes |
|---------|------------|---------------------|
| Abstractions | Everyone building agents | Rarely |
| Orchestration | People implementing multi-agent pipelines | Occasionally |
| Infrastructure | This specific dashboard app | Frequently |

## Teaching Value

This structure maps directly to how you'd explain multi-agent orchestration in a lecture:

1. **Start with Abstractions** вЂ” "What is an agent? It has a role, configuration, and an interface to run prompts."
2. **Show the Orchestration** вЂ” "Here's how multiple agents coordinate: plan, parse, execute in phases, verify with build, fix errors, and report."
3. **Build your own** вЂ” "Implement `IAgentClient` with your LLM. Implement `IWorkspace` for your file strategy. The pipeline handles the rest."

## Package Usage Examples

### Use just the Abstractions (build your own orchestration)
```xml

```

### Use the full pipeline (add your IAgentClient)
```xml

```

### Use everything (includes template client and workspace management)
```xml

```

## Key Design Decision: IWorkspace Interface

The `OrchestrationService` previously depended directly on `WorkspaceManager`. By introducing `IWorkspace`, the orchestration engine became testable and reusable without any file-system coupling. Users can provide their own workspace strategy (cloud storage, in-memory, containers, etc.).
