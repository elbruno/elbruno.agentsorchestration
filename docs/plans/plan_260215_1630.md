# Implementation Plan: Hybrid API Pattern (Option 4)

**Date**: February 15, 2026, 16:30  
**Status**: In Progress  
**Goal**: Simplify the developer experience by providing both static factory methods and DI extensions

## Problem Statement

Current usage requires developers to manually instantiate 5+ objects just to create an `OrchestrationService`:

```csharp
var workspaceRoot = Path.Combine(Path.GetTempPath(), "orchestration-simple");
var store = new AgentConfigurationStore(InstructionLoader.LoadInstructions());
var client = new TemplateAgentClient();
var factory = new AgentFactory(store, client);
var workspaceManager = new WorkspaceManager(workspaceRoot);
var service = new OrchestrationService(factory, workspaceManager);
```

This is verbose, not discoverable, and error-prone for new users.

## Solution: Hybrid Approach

Implement **both** static factory methods (for simple scenarios) and DI extensions (for ASP.NET/Aspire apps):

### Simple Console Apps

```csharp
var service = OrchestrationService.Create();
```

### ASP.NET/Aspire Apps

```csharp
services.AddOrchestration();
```

### With Configuration

```csharp
var service = OrchestrationService.Create(options => {
    options.WorkspaceRoot = "custom-path";
    options.MaxFixAttempts = 5;
});
```

**Location**: `src/ElBruno.AgentsOrchestration.Orchestration/Orchestration/OrchestrationOptions.cs`

public sealed class OrchestrationOptions
{
    public string WorkspaceRoot { get; set; } = 
        Path.Combine(Path.GetTempPath(), "orchestration-workspaces");
    public int MaxFixAttempts { get; set; } = 3;
    public IAgentClient? CustomClient { get; set; }

## Migration Guide

### Before (Old Way)

```csharp
var workspaceRoot = Path.Combine(Path.GetTempPath(), "orchestration-simple");
var store = new AgentConfigurationStore(InstructionLoader.LoadInstructions());
var client = new TemplateAgentClient();
var factory = new AgentFactory(store, client);
var workspaceManager = new WorkspaceManager(workspaceRoot);
var service = new OrchestrationService(factory, workspaceManager);
```

### After (New Way - Simple)

```csharp
var service = OrchestrationService.Create();
```

### After (New Way - With Options)

```csharp
var service = OrchestrationService.Create(options => {
    options.WorkspaceRoot = Path.Combine(Path.GetTempPath(), "my-workspace");
    options.MaxFixAttempts = 5;
});
```

### After (New Way - DI for ASP.NET)

```csharp
// In Program.cs
services.AddOrchestration(options => {
    options.WorkspaceRoot = configuration["WorkspacePath"];
});

// In your controller/hub
public class MyController
{
    private readonly OrchestrationService _service;
    
    public MyController(OrchestrationService service)
    {
        _service = service;
    }
}
```

## Backward Compatibility

✅ **Fully backward compatible** - The existing constructor on `OrchestrationService` remains unchanged. Existing code continues to work.

## Benefits

1. **90% reduction in boilerplate** for simple scenarios
2. **Idiomatic .NET patterns** - static factories for simple cases, DI for complex apps
3. **Progressive disclosure** - simple by default, powerful when needed
4. **Future-proof** - Easy to add new configuration options
5. **Discoverability** - IDE autocomplete shows `OrchestrationService.Create()`
6. **Default to Copilot SDK** - Best practice out of the box
